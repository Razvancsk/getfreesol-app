What “claim transfer” means

Instead of you directly sending tokens to users, you:

Deposit tokens into a smart contract (or backend escrow).

The contract records who can claim what (and maybe until when).

Each user calls claim() to receive their tokens.

That’s how Solana airdrops, reward programs, and vesting systems work.

⚙️ Option 1 — Simple “Claim Transfer” Contract (for a few users)

This is the easiest structure if you just want people to claim specific token amounts.

🧩 Concept

You hold all tokens in a distributor account (escrow).

Each user has an assigned amount in your on-chain state.

When they call claim(), it transfers their tokens from escrow to their wallet.

🧠 Flow

You deploy a Claim Program (custom or open source).

You store a record like:

{
  "recipient": "UserPublicKey",
  "amount": "1000000000", // 1 token if 9 decimals
  "claimed": false
}


When a user clicks “Claim,” the program:

Checks they’re the valid recipient.

Verifies they haven’t claimed before.

Transfers their tokens from escrow to their ATA.

After claim → mark as claimed → user gets tokens.

⚙️ Option 2 — Merkle Distributor (for many users)

If you have a big list (hundreds or thousands), you should use a Merkle Distributor system — same idea used by projects like Jito, Bonk, or Jupiter.

🧠 How it works

You create a CSV with {wallet, amount}.

Generate a Merkle root and deploy it to a distributor contract with your tokens.

Each user provides their Merkle proof when calling claim().

Contract checks the proof, marks them as claimed, and sends tokens.

✅ Advantages:

Cheap and scalable.

You can publish only one Merkle root on-chain (not each user record).

⚙️ Option 3 — Signed “Claim Voucher”

You (the owner) sign an off-chain message giving permission to claim a certain amount.
Your program verifies the signature and transfers tokens when the user presents it.

Example voucher fields:

{
  "recipient": "UserPublicKey",
  "amount": "1000000000",
  "nonce": 42,
  "expiry": 1736764800
}


This is great if:

Claims are on-demand (not fixed list).

You want to set expiry or anti-replay controls.

🔐 Token transfer logic

All claim methods rely on the SPL Token Program.
To transfer tokens securely:

Tokens are stored in a PDA (Program Derived Address) owned by your claim program.

The program signs a CPI (Cross-Program Invocation) to transfer tokens using spl_token::transfer.

💰 Example: Solana “claim transfer” instruction (Rust outline)
pub fn claim(ctx: Context<Claim>, amount: u64) -> Result<()> {
    let claimant = &ctx.accounts.claimant;
    let distributor = &mut ctx.accounts.distributor;
    let token_program = &ctx.accounts.token_program;
    let mint = &ctx.accounts.mint;
    let escrow = &ctx.accounts.escrow_token_account;
    let destination = &ctx.accounts.destination_token_account;

    // Check not already claimed
    require!(!distributor.claimed.contains(&claimant.key()), CustomError::AlreadyClaimed);

    // Transfer tokens
    let cpi_accounts = Transfer {
        from: escrow.to_account_info(),
        to: destination.to_account_info(),
        authority: distributor.to_account_info(),
    };
    let signer_seeds = &[b"distributor".as_ref(), &[distributor.bump]];
    let cpi_ctx = CpiContext::new_with_signer(token_program.to_account_info(), cpi_accounts, &[&signer_seeds[..]]);
    token::transfer(cpi_ctx, amount)?;

    // Mark as claimed
    distributor.claimed.push(claimant.key());

    Ok(())
}


This lets users call claim() and receive tokens directly into their wallet.

🧭 Recommended setup for your project

Since your project (GetFreeSol) already tracks wallets and balances:

Add a claim state table in your backend or Solana account.

When you run a reward cycle, move SOL or SPL tokens into a claim vault.

Let users press “Claim Rewards” in your dApp → call claim() on-chain.

Add a claim window (like your “Reclaim Period” idea) if you want them to expire.

✅ Summary
Goal	Recommended System
Few users / manual rewards	Simple Claim Contract
Many users (airdrop)	Merkle Distributor
Dynamic per-user rewards	Signed Voucher Claim
Add expiry or reclaim	Include deadline field in claim state
